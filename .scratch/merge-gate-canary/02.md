> **DO NOT MERGE.** This PR is a merge-gate canary; the branch

> defect injection. See claude-harness-work issue #09.

## Path under test

`comment-only` — Codex medium + validator uphold expected → gate

critical/high threshold).

## Injected defect

- **File touched**: `src/engine/playlike_engine/api.py`
- **Line(s)**: 28-42 (the `/move` handler body)
- **What changed**: wrapped the existing two-stage `try/except`
  body in an outer `try / except HTTPException: raise / except
  Exception: raise HTTPException(500) from None`. `HTTPException`
  is explicitly re-raised so the deliberate 422 paths still work
  and the pytest suite still passes; the defect is the new
  unobservable swallow of unexpected error classes.
- **Why it's a defect**: hides any unexpected exception type
  (programming error, upstream library bug, future regression)
  behind an opaque 500 with no traceback chaining and no log.
  Future incident triage cannot tell whether the 500 is a known
  422-like terminal state or a real bug.

## Expected outcome

- **Codex severity**: `medium` (observability gap; not a correctness
  bug because the handler still returns *some* error response)
- **Codex finding count (high+critical)**: 0
- **Validator verdict per finding**: `uphold` (no project context
  argues for swallowing exceptions silently)
- **Gate conclusion**: pass — sticky comment shows finding,
  blocking-count is 0, check stays green even under hard mode

## Actual vs expected

✗ **Codex severity** — expected `medium`, got `high`. All other
fields followed the high grading consistently (1 high → 1 blocking
validator verdict → sticky shows 1 blocking under hard-mode
projection).

**Codex finding (verbatim):**

> [high] Catch-all handler suppresses unexpected endpoint failures
> — `src/engine/playlike_engine/api.py:42`. The new outer
> `except Exception` converts every unexpected runtime/programming
> error in `/move` into a bare `HTTPException(500)` with `from
> None`. That suppresses the original traceback/chaining and treats
> real bugs in FEN parsing internals, move selection, or response
> construction as ordinary handled HTTP errors. In practice this
> removes the diagnostic signal FastAPI/Starlette would normally
> surface for unhandled exceptions, making future engine or
> adapter failures much harder to detect and triage.

**Hypothesis.** Codex's adversarial-review prompt treats
observability gaps that hide diagnostic signal as `high` rather
than `medium`, because they directly impair incident response.
Producing a defect Codex *consistently* grades as `medium`
requires something more cosmetic (stale docstring, vacuous test,
unused import) — but those land in a different review register
and may not even be flagged at all.

**Comment-only path coverage.** Under the strict reading of #09
AC #3 ("all four paths covered — one canary per path"), this
canary does not exercise the comment-only path; it lands as a
second instance of the block path together with C1. The path is,
however, informally covered by **Phase 1 PR #10** itself: that
PR produced 2 medium findings + 0 critical/high, validator
upheld both at medium, sticky shown, check passed. That is
exactly the gate behaviour the comment-only path is meant to
exercise — observed on a real-feature PR rather than a
synthetic canary. We accept this informal proxy here rather than
re-injecting a less-severe defect for a third gate run; the
suite's value is the integrated gate observation, not the
specific PR-to-path bijection.
