> **DO NOT MERGE.** This PR is a merge-gate canary; the branch

> defect injection. See claude-harness-work#23 Dimension B.

## Path under test

`comment-only` — Codex grades the defect `medium` (silent
failure / observability gap), validator upholds (`setError` is
the file's loud-on-failure pattern; `apiClient.ts` also surfaces
errors loudly), gate passes because severity-floor filters
`medium` out of `critical|high ∩ uphold`.

Category: **observability** (Dim B #3). Phase 2 surface exposes
this category through `useChessSession.ts:96–106` — the
illegal-UCI catch block calls `setError(new ApiError(...))`
explicitly so operators and users get a signal when the engine
returns a malformed move.

## Injected defect

- **File touched**: `src/web/app/hooks/useChessSession.ts`
- **Line(s)**: removed line 103 (was
  `setError(new ApiError(0, \`engine returned illegal move "${uci}"\`))`).
- **What changed**: a single-line deletion. The illegal-UCI
  catch branch still rolls back chess.js, syncs `fen`, slices
  `history`, and clears `awaiting` — but the *signal* that
  something went wrong is gone. The error surface is silenced
  while the recovery sequence runs cleanly.
- **Why it's a defect (the case for `uphold`)**: The original
  code is loud-on-purpose. After the deletion:
  - chess.js is rolled back (correct).
  - React `fen` reflects the rollback (correct).
  - `history` loses the user's move (correct).
  - `awaiting` clears (correct).
  - **But `error` stays at whatever it was before** — typically
    `null`. The UI shows no error banner, the operator sees no
    log signal, and from the user's perspective the move simply
    "didn't happen". A backend bug returning malformed UCI is
    invisible.
  Surrounding contract: `setError(new ApiError(...))` is the
  file's documented pattern for surfacing rejection (used in the
  422 branch too); `apiClient.ts` also surfaces errors loudly
  by raising `ApiError` rather than swallowing them. The
  deletion breaks this two-citation pattern.

- **Gate conclusion**: `pass` (no `critical|high ∩ uphold`;

## What this attempt measures

The `comment-only` path was already verified end-to-end on Dim A
C2 (`07.md`, stale TODO → low → uphold → pass). #11 re-exercises
the same gate path on a *different kind* of finding: silent
failure / observability gap rather than cosmetic staleness. The
free observation is that the severity-floor design works the
same for two distinct severity bands (`low` in C2, `medium`
here).

The Dim B bar is **category coverage**, not exact path
prediction. An `observability`-category injection that fires as
`high` and blocks is still a category success — what matters is
that the gate's response to a silent-failure defect is observed
end-to-end.

A real miss would be: Codex returns 0 findings (no flag at all)
or grades the defect non-observability in character (e.g.
labels it dead-code removal, unused import, or refactoring
opportunity). In that case, propose a second observability
injection per the ≥2-attempts rule.
