# Canary 03 — insecure RNG for "engine session token"

> **DO NOT MERGE.** This PR is a merge-gate canary; the branch
> existed solely to observe how the gate responds to a deliberate
> defect injection. See claude-harness-work issue #09.

## Path under test

`validator-dismiss` — Codex high on insecure RNG + validator
dismiss with citation to `src/web/CONTEXT.md` (Session is
gameplay-only, owned by browser, never persisted server-side)
expected.

## Injected defect

- **File touched**: `src/engine/playlike_engine/api.py`
- **Line(s)**: 19-25 (new helper) + 36-37 (call inside `/move`)
- **What changed**: added a `_new_engine_session_token()` helper
  using `random.choice(string.ascii_letters + string.digits)`
  over 16 characters, called from `/move` and logged for
  "client-side request correlation". Imports `random`, `string`,
  `logging`.
- **Why it would be a defect**: the phrasing "session token" is
  deliberately security-adjacent — `random` is a non-cryptographic
  PRNG and the canonical advice is to use `secrets.token_hex`/
  `secrets.token_urlsafe` for anything named like a session token.
- **The dismiss the validator was meant to find**: `src/web/CONTEXT.md`
  defines Session as a gameplay-only construct, owned by the
  browser, never persisted server-side. The Engine is stateless
  per request and has no security session concept. The "token" in
  this canary is purely an internal log correlator — no client
  ever sees it, no authentication is gated on it. Under that
  reading, `random` is appropriate and `secrets` is overkill.

## Expected outcome

- **Codex severity**: `high` (insecure RNG for security-adjacent token)
- **Codex finding count (high+critical)**: ≥ 1
- **Validator verdict per finding**: `dismiss` with citation to
  `src/web/CONTEXT.md` (Session is gameplay-only, browser-owned)
- **Gate conclusion**: pass — sticky shows the finding, validator
  layer reports 0 blocking verdicts after dismiss

## Actual vs expected

✗ **Codex severity** — expected `high`, got `low` / `medium` /
`medium+low` across three runs. Codex never reached high and
never framed the issue as "insecure RNG"; all three runs
classified the helper as an observability/correctness issue
(token is generated but not exposed to the client, so it cannot
actually serve correlation).

**Hypothesis.** Codex's adversarial-review prompt does its own
context reading and correctly observed that this codebase has
no security boundary in the Engine — no Set-Cookie, no auth
middleware, no header exposure of the token. Under that
observation, `random` is not graded as a security finding even
when the helper is named `_new_engine_session_token`. Codex is
in effect doing the same context reasoning the validator was
*meant* to do — just one layer earlier and with the opposite
conclusion (it doesn't flag rather than flagging+dismissing).

**Validator-dismiss path coverage: unmet, and structurally hard
to test in this codebase.** To produce a Codex finding that
would naturally invite a CONTEXT.md-based dismiss, we would need
to engineer the injection until Codex's heuristic fires — e.g.,
rename to `_new_engine_auth_token` and actually expose the value
in a response header. But that would (a) cross into "engineering
Codex's prompt rather than testing the gate", and (b) probably
flip the validator to `uphold` because the validator is also
reading the same project context and would conclude `auth` has
no documented dismiss-grounds in the Engine. **We accept the
coverage gap rather than fabricate a finding to dismiss.**

**Variance spot-check.** Severity drifted across the three runs
(low → medium → medium+low). The *finding* itself was stable
(token-not-exposed-to-client framing every time), but the
severity grade was not deterministic. This is a property worth
noting for #10's measurement framing — Codex's grade is not
reproducible at the severity-bucket level on subtle findings,
even when the diff is byte-identical. For high/critical the
buckets are probably more stable (C1 ran once and would
presumably be high on rerun) but that is unverified by this
suite.

## Follow-up implication for the open `file:line` cross-check
decision

STATUS's open-decisions section flagged a concern that
`parse_validator_output` could be tricked by a `dismiss` token
appearing in a validator's prose. C3's intent was to exercise
the natural dismiss path so that concern could be re-evaluated
with real signal. Since C3 did not actually produce a dismiss
verdict, the cross-check decision remains in the same evidential
state as before — it should be re-evaluated when a real dismiss
naturally appears in production traffic, not on a synthetic
canary.
