# Canary 09 — Dim B data-loss: missing chess.js undo() in engine-error rollback

> **DO NOT MERGE.** This PR is a merge-gate canary; the branch
> exists solely to observe how the gate responds to a deliberate
> defect injection. See claude-harness-work#23 Dimension B.

## Path under test

`block` — Codex grades the defect `high` or `critical` (missing
rollback / state corruption), validator upholds citing the
Invariant-5 comment block, gate blocks under hard mode
(`critical|high ∩ uphold ≥ 1`).

Category: **data-loss** (Dim B #1). Phase 2 surface exposes
this category through `useChessSession.ts:74–87` — the
engine-error rollback block calls `chessRef.current.undo()`
explicitly to keep chess.js, React state, and history in sync.

## Injected defect

- **File touched**: `src/web/app/hooks/useChessSession.ts`
- **Line(s)**: removed line 82 (was `chessRef.current.undo()`).
- **What changed**: a single-line deletion. The engine-error
  branch in `requestEngineMove` no longer rolls back chess.js
  state. `setFen(chessRef.current.fen())` still runs, but now
  observes the *unrolled* internal chess.js board, so the
  user's just-attempted move is committed to chess.js while
  React `history` is sliced back by one and `error` is set.
  Three pieces of state diverge.
- **Why it's a defect (the case for `uphold`)**: The block sits
  directly under the comment naming "Invariant 5" / 422
  defensive freeze. The whole block exists *to* roll back when
  the server rejects the user's move. After the deletion:
  - chess.js retains the user's move.
  - React `fen` reflects that retained-move position.
  - `history` loses one entry (slice -1).
  - `awaiting` clears, `error` is set.
  Next user move proceeds against a position the server never
  agreed to — and chess.js's notion of legal moves now differs
  from the server's. Classic missing-rollback data-loss /
  state-corruption defect.

## What this attempt measures

The `block` path has already been verified end-to-end (Dim A
C1 in `01.md`). #09 re-exercises it on a different *kind* of
defect — Phase 2 frontend / state-coherence rather than
Phase 1 hardcoded secret — to confirm the same path fires for
Dim B's data-loss category.

The bar is **category coverage**, not exact severity prediction.
A `high`-graded finding upheld and blocked is success; so is
`critical`. A `medium` upheld → comment-only would be a
near-miss (severity-floor active) but **not** a category miss:
the data-loss category was still observed end-to-end.

A real miss would be: Codex returns 0 findings (no flag at all)
or grades the defect non-data-loss in character (e.g. labels
it cosmetic UI). In that case, propose a second data-loss
injection per the ≥2-attempts rule.
