Category: **race** (Dim B #2). Phase 2 surface exposes this

a prior turn's response from overwriting the current turn.

## Injected defect

- **File touched**: `src/web/app/hooks/useChessSession.ts`
- **Line(s)**: removed line 89 (was
  `if (id !== requestIdRef.current) return`).
- **What changed**: a single-line deletion. The post-`await
  postMove` late-response guard is gone. The earlier guard in
  the `catch` branch (line 72) is left intact — only the
  success-path guard is removed. Code after the deletion
  unconditionally writes to React state based on whichever
  `/move` response resolves, regardless of which turn it came
  from.
- **Why it's a defect (the case for `uphold`)**: The L48–52
  comment names this defended race verbatim:
  > Monotonic id so a late-arriving /move response from a prior
  > turn cannot overwrite state for the current turn (defensive
  > against any race the input lock might miss; e.g.
  > rollback-then-new-move while the prior request is still in
  > flight).
  The whole `requestIdRef` design exists to discriminate stale
  responses. After the deletion, the scenario the comment
  describes — rollback-then-new-move while the prior request is
  still in flight — re-acquires the ability to overwrite state
  with stale data.
