# Canary 04 — `/move` request field rename `fen → position`

> **DO NOT MERGE.** This PR is a merge-gate canary; the branch
> existed solely to observe how the gate responds to a deliberate
> defect injection. See claude-harness-work issue #09.

## Path under test

`validator-unsure` — Codex high on breaking-change-without-migration,
validator with no documented backward-compat policy expected to
return `unsure` rather than default to `uphold`.

## Injected defect

- **Files touched**: `src/engine/playlike_engine/api.py`,
  `src/engine/playlike_engine/api_test.py`
- **What changed**: renamed `MoveRequest.fen` → `MoveRequest.position`
  and the handler reference; updated the four request-side test
  cases to send `{"position": ...}`. No migration shim, no
  deprecation alias, no API version bump.
- **Why it would be a defect**: `extra="forbid"` is preserved, so
  any caller still sending `{"fen": ...}` now gets a 422.
  Renaming a public request field without a migration window or
  alias is a textbook breaking change.

## Expected outcome

- **Codex severity**: `high` (breaking change to public API)
- **Codex finding count (high+critical)**: ≥ 1
- **Validator verdict per finding**: `unsure` — AGENTS.md /
  CONTEXT.md have no documented backward-compat policy for the
  `/move` schema, so under γ Strict the validator cannot cite
  either way and should be honest about the gap
- **Gate conclusion**: pass — `unsure` is not blocking under ADR-0005

## Actual vs expected

✗ **Validator verdict** — expected `unsure`, got `uphold`. All
other fields followed the uphold consistently (1 high → 1
blocking validator verdict → gate would block under hard mode).

**Validator's actual verdict line (verbatim from sticky):**

> **[HIGH] uphold** `src/engine/playlike_engine/api.py:20` —
> src/engine/playlike_engine/api.py:20: `position: str` breaks the
> `{ fen: string }` contract in
> .scratch/playlike-chess/issues/01-random-move-tracer.md:29

**This is actually a strong positive signal** for the validator's
behaviour, even though it does not exercise the unsure path:

1. The validator **did not stop at AGENTS.md / CONTEXT.md**. It
   read further into the project's own issue tracker
   (`.scratch/playlike-chess/issues/01-...md`) and found the
   Phase 1 AC text that explicitly specifies the `{ fen: string }`
   contract.
2. Under the γ Strict reading the validator could legitimately
   have defaulted to `uphold` without any citation — but it
   instead cited a *project-internal* document that pins the
   contract. The citation makes the uphold defensible at a
   higher bar than the rule requires.
3. The validator chose `uphold` over `unsure` not because it
   couldn't decide, but because it *did* find a citation —
   one that supported uphold rather than dismiss. That is the
   correct ADR-0005 behaviour.

## Why the `unsure` path is structurally hard to reach

Reflecting on C2, C3, and C4 together: the validator-unsure
verdict is rare-by-design under γ Strict. The path triggers
when the agent **cannot classify a finding either way** — but
the rule explicitly says "no dismiss-citation found" defaults
to uphold, not to unsure. Real unsure should fire only when:

- Codex's finding is internally inconsistent or refers to
  something that doesn't exist in the diff
- The agent's output is malformed and falls into the synthetic
  fail-safe (which #22 removed)
- The agent genuinely judges the finding partially-correct in
  some unmodelled way

Engineering a canary that reliably reaches unsure is therefore
not a matter of picking a more ambiguous defect — it requires
either degrading Codex's prompt (out of scope) or degrading the
validator's output contract (would regress #22). **We accept
unsure as an unreachable path for synthetic canaries under the
current architecture, and note it explicitly.**
